# Software Specification
A system provided as an app and a set of station kiosks, which is used by passengers who want to report issues and incidents related to the Copenhagen Metro (CPH Metro). The passengers travel with the metro on their own initiative, and their only relation to the system owner (Metro Service A/S) is that they use the app or a kiosk to submit and track issue reports. The system is primarily an administrative tool that is responsible for registering all reported issues (e.g., broken ticket machines, malfunctioning escalators, cleanliness problems, safety concerns), assigning them to the appropriate maintenance or security teams, and tracking their status until resolution. Secondarily, it is a communication medium that passengers use to receive feedback and status updates on their reports, and that metro staff use to request additional information or notify passengers about completed actions. The system can register a new passenger profile (optionally anonymous), create an issue report with location (station, platform, train), category, description, and photos, update the status of an issue as it progresses through different handling stages, and notify passengers about changes to the status of their reports. The app is running on each passenger’s smartphone, while kiosk clients are installed at selected metro stations. Both app and kiosks communicate through a secure network with a central server that maintains all issue reports, assignments, and audit logs. On the smartphone, there is always a cached copy of the passenger’s open and recently closed reports from the current day, so they are accessible if the mobile data network is temporarily unavailable. The app includes support for GPS-based station suggestions and a QR code scanner to identify specific assets (e.g., ticket machines or information screens) where available. The system will be developed by Metro Service A/S’s IT department in cooperation with operations, maintenance, security staff, and a small group of representative passengers. It may be necessary to resolve conflicting requirements between these different stakeholder groups (e.g., level of detail vs. ease of use vs. privacy). The app and kiosks will be used by users with very different levels of IT skills and varying familiarity with the English and Danish languages.
# Functional Requirements
## User Management
FR1: The user should be able to register a passenger profile that is optionally anonymous.
## Issue Tracking
FR2: The passenger should be able to create an issue report with location, category, description and photos.
FR3: The passenger should be able to scan a QR code to identify specific assets.
FR4: The passenger should be able to view station suggestions based on GPS location.
FR5: The passenger should be able to be notified when there are changes in status of their reports.
## Data Access
FR6: The passenger should be able to view open and recently closed reports from the current day, even if offline.
